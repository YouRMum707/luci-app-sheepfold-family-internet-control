# Кэш браузера LuCI и версионирование ассетов

Sheepfold не должен просить пользователя чистить кэш браузера после каждого обновления LuCI-интерфейса.

## Правило

Нужна одна проектная версия LuCI-ассетов, которая добавляется ко всем URL JS/CSS/static ресурсов Sheepfold:

```text
/luci-static/resources/sheepfold/app.css?v=0.1.0-1
/luci-static/resources/sheepfold/client-table.js?v=0.1.0-1
```

Канонический источник версии — версия OpenWRT-пакета:

```make
PKG_VERSION:=0.1.0
PKG_RELEASE:=1
SHEEPFOLD_UI_ASSET_VERSION:=$(PKG_VERSION)-$(PKG_RELEASE)
```

Итоговое runtime-значение:

```text
ui_asset_version = PKG_VERSION-PKG_RELEASE
```

В UCI-конфиге по умолчанию значение можно хранить так:

```text
option ui_asset_version '0.1.0-1'
```

Package `postinst` должен записывать `SHEEPFOLD_UI_ASSET_VERSION` в UCI при установке/обновлении. В будущей реализации можно генерировать небольшой LuCI config module на этапе сборки пакета, а не полагаться только на UCI default. Главное правило: разработчик поднимает версию в одном месте для релиза, и все URL LuCI-ассетов получают новый cache-busting suffix.

## Требования

- Не хардкодить разные версии ассетов в отдельных LuCI-файлах.
- Использовать один `ui_asset_version` для всех Sheepfold LuCI CSS, JS, картинок, шрифтов и дополнительных static resources.
- Поднимать версию, когда меняются frontend-файлы LuCI.
- Скрипт обновления должен перезапустить/перезагрузить Sheepfold service после обновления пакета и может показывать подсказку по очистке кэша только как запасной вариант.
- Ссылку Podkop про очистку кэша браузера оставить в README/docs как troubleshooting, а не как нормальный сценарий обновления.

## Ограничение

Главный LuCI view loader всё равно может зависеть от поведения кэша LuCI/OpenWRT в конкретной сборке. Версионирование наших подключаемых ассетов уменьшает обычную проблему, но в troubleshooting всё равно стоит оставить ручную очистку кэша для редких случаев, когда браузер показывает старый view.
