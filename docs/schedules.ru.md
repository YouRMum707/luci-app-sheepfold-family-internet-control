# Расписания доступа

Расписания — один из основных сценариев Sheepfold.

Интерфейс должен быть понятным родителям и похожим по духу на простые правила доступа: выбрать устройство, дни недели и время.

## Основные возможности

Правило расписания должно включать:

- название правила;
- список устройств или группу устройств;
- дни недели;
- время начала;
- время окончания;
- действие: разрешить или заблокировать;
- состояние: включено/выключено;
- короткое описание для родителя.

## Приоритеты

Расписания применяются только к устройствам, которых нет в белом и чёрном списках.

Приоритет:

1. Чёрный список.
2. Белый список.
3. Временное разрешение.
4. Расписание.
5. Общие правила.

Временное разрешение может временно перекрыть расписание, но никогда не перекрывает чёрный список.

## Время

Нужно поддержать интервалы, пересекающие полночь.

Пример:

```text
22:00 -> 07:00
```

Такое правило означает вечер текущего дня и утро следующего дня.

## Конфликты

Если два расписания конфликтуют, интерфейс должен показать предупреждение.

Backend должен иметь детерминированное поведение:

- явная блокировка сильнее явного разрешения в расписаниях одного уровня;
- более конкретное правило устройства сильнее группового правила;
- при равенстве применяется более новое или более приоритетное правило, но это должно быть видно в интерфейсе.

## UI

В LuCI и Android нужны:

- список правил;
- быстрый переключатель правила;
- редактор дней недели;
- редактор времени;
- предпросмотр "что будет сейчас";
- подсказка, почему устройство сейчас заблокировано или разрешено.

## Логи

В журнал писать только факт срабатывания расписания:

```text
schedule applied: rule name, target device display name, action, result
```

Не писать историю сайтов, DNS-запросы или лишние технические данные.
